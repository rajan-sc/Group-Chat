# Nexus Chat Technical Documentation

This document provides comprehensive, enterprise-grade technical documentation for the Nexus Group-Chat application. It covers high-level architecture, line-by-line file logic, directory structures, database schema, real-time networking, and developer guidelines.

---

## 1. Project Overview

### What the project does
Nexus Chat is a full-stack, real-time chat application. It allows users to register, log in, join a default "Global Chat", communicate in direct messages or custom rooms, share files via AWS S3, and utilize Google Gemini AI for smart reply generation and message autocompletion.

### Main Purpose
To provide a highly scalable, real-time messaging platform demonstrating advanced websocket communication, secure authentication, external cloud storage integration, and AI-assisted UX paradigms.

### Core Features
- **Real-Time Websockets:** Instant messaging across multiple rooms without page reloads.
- **AI Integration:** Google Gemini integration for smart replies and autocomplete.
- **Cloud Storage:** Secure file uploads securely backed by AWS S3.
- **Cron Jobs:** Automated database archival running daily at midnight to maintain performance.
- **Premium UI:** Glassmorphism-inspired design with a unified routing interface.

## 2. Directory Structure

The project follows a standard MVC (Model-View-Controller) plus Services architectural pattern, optimizing for scalability and separation of concerns.

```text
Group-Chat/
├── app.js                   # Application entry point (Server, Express, Sockets)
├── models/                  # Database schema definitions (Sequelize)
├── controllers/             # Request handlers (Business Logic)
├── routes/                  # Express route definitions
├── middleware/              # Authentication and validation layers
├── sockets/                 # Socket.io event handlers
├── services/                # External API integrations (AWS S3, Gemini)
├── jobs/                    # Scheduled cron jobs
├── utils/                   # Shared utilities (DB connection)
├── public/                  # Static frontend assets (CSS, Vanilla JS)
└── views/                   # HTML files served to the client
```

## 3. Core Modules & Linking Logic

### 3.1. Application Entry (`app.js`)
`app.js` is the central nervous system. It initializes the Express application, configures CORS, parses JSON bodies, and serves static files from `/public`. It then connects the HTTP server to `Socket.io` and mounts the API routes (`/api/auth`, `/api/chat`, `/api/ai`). Finally, it initializes the database connection and scheduled Cron Jobs before listening on port 3000.

### 3.2. Models (Database Layer)
Managed via `Sequelize` ORM connecting to a MySQL instance.
- **`user.js`:** Defines the `User` table (`id`, `name`, `email`, `password`).
- **`message.js`:** Defines the active messages (`roomId`, `roomName`, `senderId`, `message`, `fileUrl`). Belongs to a User.
- **`archivedMessage.js`:** An exact clone of the `Message` table used for cold storage.
- **`index.js`:** Defines the database relationships (e.g., `User.hasMany(Message)`).

### 3.3. Controllers (Business Logic)
Controllers receive the `req` and `res` objects from routes, interact with Models or Services, and return JSON.
- **`authController.js`:** 
  - `register`: Hashes passwords using `bcrypt` and creates a User.
  - `login`: Compares passwords and generates a `jsonwebtoken` containing `id`, `email`, and `name`.
- **`chatController.js`:** 
  - `uploadFile`: Receives a file from multer, pipes it to `awsS3Service.uploadToS3()`, and returns the public URL.
- **`aiController.js`:**
  - Extracts chat context and forwards it to `geminiService.js`.

### 3.4. Services (External Integrations)
- **`awsS3Service.js`:** Uses `@aws-sdk/client-s3` to upload binary buffers to an AWS bucket securely.
- **`geminiService.js`:** Uses `@google/genai` to inject conversation history and retrieve LLM-generated string responses for autocomplete and smart replies.

### 3.5. WebSockets (`sockets/chatSocket.js`)
Real-time communication bypassing HTTP overhead.
- **Linking:** Mounted in `app.js` and protected by `socketAuth.js`.
- **`join_room`:** Subscribes a socket to a specific room ID (generated by hashing target emails or custom names) and fetches `previous_messages` from the DB using `findAll`.
- **`send_message`:** Writes a new message to the DB, then broadcasts `receive_message` to the active room.
- **`disconnecting`:** Iterates through a user's active rooms to broadcast a system message: `[Name] joined.` or `[Name] has left the chat.`

### 3.6. Cron Jobs (`jobs/archiveJob.js`)
Uses `node-cron`. Runs `0 0 * * *` (midnight). It wraps operations in a SQL Transaction: selects all data from `Message`, bulk inserts it into `ArchivedMessage`, and then `destroy`s it from `Message`. Ensures the active table stays lean and fast.

## 4. Authentication & Middlewares

### 4.1. HTTP API Middleware (`middleware/authMiddleware.js`)
**How it works:** Intercepts API requests (like file uploads or AI generation) to verify the `Authorization: Bearer <token>` header.
**Authentication:** Uses `jsonwebtoken.verify` against `process.env.JWT_SECRET`. If valid, attaches the decoded payload to `req.user`.
**Limitations:** 
- Uses stateless JWTs meaning tokens cannot be easily revoked before expiration (24h) unless a blacklist database is implemented.
- Dependent entirely on the client storing the token securely (currently in `localStorage`).

### 4.2. Socket Middleware (`middleware/socketAuth.js`)
**How it works:** Socket.io allows custom middleware via `io.use()`. 
**Authentication:** Intercepts the socket connection handshake (`socket.handshake.auth.token`), verifies the JWT, and attaches the payload to `socket.user`.
**Limitations:**
- Authentication is only performed on connection. If the JWT expires while the socket remains open, the user retains access until they refresh the page.

## 5. Frontend Architecture

Located in `public/` and `views/`. Uses Vanilla JavaScript to avoid build steps while maintaining high performance.

### 5.1. Authentication Flow (`public/js/auth.js`)
Handles the toggling between login and register forms. Extracts the user's Name, Email, and Password, sending them via `fetch` API. Upon successful login, it saves the `token`, `name`, and `email` to `localStorage` and redirects to `/chat`.

### 5.2. Real-Time Chat Controller (`public/js/chat.js`)
The heaviest client-side logic.
- **Initialization:** Reads local storage. Disconnects unauthorized users. Connects to `io()`.
- **UI Management:** Switches active UI states between Global Chat, Custom Rooms, and Direct Messages.
- **Rendering (`addMessageToUI`):** Dynamically constructs DOM elements to display messages. Differentiates `isSelf` via `msg.senderEmail === userEmail` to style bubbles correctly. Applies unique styles to system messages.
- **Smart Features:** Handles typing indicator timeouts and interacts with the AI endpoints to manipulate the message `placeholder` to `"Thinking..."` dynamically.

## 6. Security Limitations & Future Improvements

1. **XSS Vulnerability (Cross-Site Scripting):** Tokens are stored in `localStorage`, making them accessible to any malicious scripts injected into the DOM. 
   - *Fix:* Move JWTs to `httpOnly` secure cookies.
2. **Rate Limiting:** Authentication and AI endpoints currently lack rate limiters, leaving them open to brute force and quota exhaustion.
   - *Fix:* Implement `express-rate-limit`.
3. **Database Constraints:** `User` names are forced to be non-null. Existing rows without names will break during migrations if not handled.
4. **Error Handling:** Socket errors are currently logged to the console but lack robust retry mechanisms or dead-letter queues.
